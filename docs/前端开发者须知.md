# ArenaQA 项目后端说明 — 给前端开发者的手册

> 你好！这是 ArenaQA（AI 问答竞技场）项目后端的完整说明。项目采用 Next.js 15 全栈架构，后端的 API、SSE 流式、数据流和状态管理都集中在同一个项目中。你现在需要写前端覆盖它。

---

## 一、项目概述

### 这是什么
ArenaQA 是一个**并发 AI 问答工作台**——用户可以同时向 DeepSeek、Qwen、Claude、Gemini 等多个 AI 模型提问，并排查看各模型的回答，然后由 AI 裁判（GPT-4o）自动评分并融合出综合答案。

### 技术栈
| 类别 | 技术 |
|------|------|
| 框架 | Next.js 15 (React 19) — App Router |
| 样式 | Tailwind CSS 4 + CSS Variables |
| 流式 | Server-Sent Events (SSE) |
| 状态 | Zustand |
| AI SDK | Vercel AI SDK |

### 核心设计思路
整个应用像是一个"模型竞技场"——
1. **用户提问** + 选择参赛模型
2. **并发发送**给所有选中的模型
3. **SSE 流式**返回每个模型的回答（逐 token 推送）
4. **AI 裁判**（GPT-4o）对每个模型做 4 维评分
5. **融合引擎**提取所有模型的共识和分歧，生成综合答案

## 二、数据流全景（非常重要）

```
┌─────────────────────────────────────────────────────────┐
│  前端 (React)                   后端 (Next.js API)       │
│                                                          │
│  1. 用户输入问题 + 选模型                                │
│     │                                                     │
│     ▼ POST /api/chat/send                                 │
│  2. 发送 { prompt, models[] } ──────────→  TaskManager    │
│     │                                                     │
│     │    ←── { taskId } ──────────────  启动并发任务      │
│     │                                                     │
│  3. 连接 EventSource(GET /api/chat/stream/{taskId})       │
│     │                                                     │
│     │    ←── event: chunk ───────────  DeepSeek 流式返回  │
│     │    ←── event: chunk ───────────  Qwen 流式返回      │
│     │    ←── event: chunk ───────────  Claude 流式返回    │
│     │    ←── event: done  ───────────  某个模型完成       │
│     │    ←── event: error ───────────  某个模型失败       │
│     │    ←── event: judge ───────────  AI 裁判评分结果    │
│     │    ←── event: fusion ──────────  融合分析结果       │
│     │    ←── event: complete ────────  全部完成           │
│     │                                                     │
│  4. 前端关闭 EventSource                                  │
└─────────────────────────────────────────────────────────┘
```

## 三、API 接口文档

### 1. POST /api/chat/send — 发起问答

**请求体：**
```json
{
  "prompt": "什么是量子计算？",
  "models": ["deepseek", "qwen", "claude"]
}
```

**成功响应 (201)：**
```json
{
  "taskId": "task_abc12345",
  "models": ["deepseek", "qwen", "claude"],
  "createdAt": "2026-06-08T12:00:00.000Z"
}
```

**错误响应：**
- 422 — prompt 为空
- 400 — models 为空或包含无效模型名称
- 500 — 内部错误

**前端操作：** 拿到 `taskId` 后，立即连接 SSE。

---

### 2. GET /api/chat/stream/{taskId} — SSE 流式连接

**协议：** Server-Sent Events (EventSource)

**连接方式：**
```javascript
const eventSource = new EventSource(`/api/chat/stream/${taskId}`);
```

**事件列表（按发生顺序）：**

#### a) event: connected
首次连接确认。
```json
{}
```

#### b) event: chunk — 流式文本块
每个模型的回答以多个 chunk 事件逐 token 推送。
```json
{
  "model": "deepseek",
  "content": "量子计算是一种",
  "index": 0
}
```
- 前端需要将同一 model 的 content **累加拼接**
- index 是递增序号，但可以忽略直接用 content 拼接

#### c) event: done — 某个模型完成
```json
{
  "model": "deepseek",
  "latencyMs": 4230,
  "totalChars": 1520
}
```

#### d) event: error — 某个模型出错
```json
{
  "model": "deepseek",
  "error": "Response timeout",
  "code": "TIMEOUT"
}
```
code 取值: `TIMEOUT` | `PROVIDER_ERROR` | `RATE_LIMITED`

#### e) event: judge — AI 裁判评分结果（所有模型全部完成后触发）
```json
{
  "scores": [
    {
      "model": "deepseek",
      "accuracy": 8,
      "completeness": 7,
      "actionability": 9,
      "safety": 10,
      "total": 8.5,
      "brief": "回答准确全面，但缺少具体示例"
    }
  ]
}
```
- 评分维度 4 个（1-10 分）：accuracy, completeness, actionability, safety
- total = 4 维平均值
- **如果没有配置 OPENAI_API_KEY，所有分为 0，brief 提示未配置**

#### f) event: judge_error — 评分失败
```json
{
  "error": "Judge API call failed"
}
```

#### g) event: fusion — 融合分析结果（评分之后触发）
```json
{
  "consensus": ["量子计算利用叠加态", "量子比特是基本单位"],
  "divergences": [
    {
      "topic": "量子霸权实现时间",
      "positions": {
        "deepseek": "10年内",
        "qwen": "可能需要20年以上",
        "claude": "尚不确定"
      }
    }
  ],
  "synthesized": "### 综合答案\n\n量子计算是一种...（Markdown 全文）"
}
```
- **consensus**：所有模型达成一致的要点列表
- **divergences**：各模型存在分歧的观点
- **synthesized**：AI 融合后的综合答案（Markdown 格式）
- **没有 OPENAI_API_KEY 时**：consensus 为提示信息，synthesized 为简单拼接

#### h) event: complete — 全部流程完成
```json
{
  "taskId": "task_abc12345",
  "totalLatencyMs": 15600,
  "allFailed": false
}
```
- 前端收到此事件后必须调用 `eventSource.close()`
- `allFailed: true` 表示所有模型都失败了

## 四、核心类型定义（前端需要了解的）

```typescript
// 模型元信息
interface ModelConfig {
  name: string;          // 标识，如 "deepseek"
  displayName: string;   // 展示名，如 "DeepSeek V3"
  providerType: 'openai_compat' | 'anthropic' | 'google' | 'playwright';
  enabled: boolean;      // 是否已配置 API Key
}

// 单个模型的回答状态
interface AnswerState {
  model: string;
  content: string;       // 累积的文本内容（流式拼接）
  status: 'streaming' | 'done' | 'error';
  latencyMs?: number;
  error?: string;
}

// 全局聊天状态
type ChatStatus = 'idle' | 'streaming' | 'judging' | 'fusing' | 'complete';

// 评分项
interface Score {
  model: string;
  accuracy: number;      // 准确性 1-10
  completeness: number;  // 完整性 1-10
  actionability: number; // 可操作性 1-10
  safety: number;        // 安全性 1-10
  total: number;         // 平均分
  brief: string;         // 一句话评价
}
```

## 五、前端关键逻辑要点

### 5.1 状态机流转

```
idle → streaming → judging → fusing → complete
                    ↓                     ↑
                (流式接收中)       (收到 complete 事件)
```

- 状态变更由 SSE 事件驱动
- `judging`：收到 `judge` 事件时进入
- `fusing`：收到 `fusion` 事件时进入
- `complete`：收到 `complete` 事件时进入

### 5.2 流式内容拼接

前端收到同一个 model 的多个 chunk 事件时，需要**累加拼接** content：
```typescript
// 伪代码 — 这个逻辑必须正确
answers[model].content += chunk.content;
```

### 5.3 全局状态 Zusutand Store

项目已经有一个完整的 Zustand store（`src/stores/chat-store.ts`），包含了：
- `answers: Record<string, AnswerState>` — 每个模型的回答
- `judgeResult` — 评分结果
- `fusionResult` — 融合结果
- `status` — 当前阶段
- `sendChat(prompt, models)` — 发送提问 + 连接 SSE
- `appendChunk(model, content)` — 追加流式文本
- `reset()` — 重置状态

**你可以直接复用这个 store**，或者在 `sendChat` 方法中找到 SSE 事件处理逻辑，对应到你的前端组件。

### 5.4 页面布局

当前采用**三栏可拖拽布局**（react-resizable-panels）：
```
┌──────────┬────────────────────────┬──────────┐
│          │                        │          │
│ 输入面板  │     多列回答区          │ 结果面板  │
│ (25%)    │     (50%)              │ (25%)    │
│          │                        │          │
│ 模型选择  │  ┌─────┬─────┬─────┐  │ AI 评分  │
│ 问题输入  │  │DeepS │Qwen │Claude│  │ 融合分析  │
│ 发送按钮  │  └─────┴─────┴─────┘  │ 置信度   │
│          │                        │          │
└──────────┴────────────────────────┴──────────┘
```

## 六、重点 UI 问题（当前不好看的地方）

你们说的"UI 不好看"主要问题出在以下方面，建议重点改进：

### 6.1 整体视觉
- ✅ **配色**：当前紫色/靛蓝主题已经可以
- ❌ **间距**：面板间距不均匀，有些地方太挤
- ❌ **一致性**：不同区域的圆角、阴影风格不统一
- ❌ **响应式**：在小屏幕上布局会崩

### 6.2 输入面板
- ❌ **模型选择**：当前是简陋的按钮列表，建议改为漂亮的卡片式切换
- ❌ **文本输入框**：太简陋，缺乏 modern 输入框的质感
- ❌ **发送按钮**：样式简单，缺少状态反馈动画

### 6.3 回答列
- ❌ **Markdown 渲染**：当前是自己写的简单 `SimpleMarkdown` 组件，只支持 `**粗体**`` `行内代码` `` ```代码块``` `` 和最基础的列表。不支持表格、图片、链接、标题层级等
- ❌ **推荐**：使用 `react-markdown` 或 `@uiw/react-markdown-preview` 替代
- ❌ **代码高亮**：当前代码块是纯色背景，建议用 `prismjs` 或 `shiki` 做语法高亮
- ❌ **流式光标**：当前闪烁光标比较简陋

### 6.4 侧边栏（结果面板）
- ❌ **评分卡片**：可以更美观，增加动画和视觉反馈
- ❌ **融合结果**：展示形式可以更丰富（折叠/展开等）

### 6.5 动画和交互
- ❌ **加载态**：页面初始加载时缺少骨架屏
- ❌ **过渡动画**：状态切换的动画不够流畅
- ❌ **响应式**：三栏布局在窄屏上应该自动切换为上下布局

## 七、环境变量配置

前端需要知道以下环境变量（实际由后端读取，但前端可以用于判断可用性）：

| 变量 | 说明 |
|------|------|
| `DEEPSEEK_API_KEY` | DeepSeek API 密钥 |
| `QWEN_API_KEY` | 通义千问 API 密钥 |
| `ANTHROPIC_API_KEY` | Claude API 密钥 |
| `GEMINI_API_KEY` | Gemini API 密钥 |
| `OPENAI_API_KEY` | OpenAI API 密钥（用于裁判评分） |
| `JUDGE_MODEL` | 裁判模型，默认 `gpt-4o` |

**关键提示：** 如果 `OPENAI_API_KEY` 未配置，AI 裁判评分和融合引擎将**不能正常工作**（评分全为 0，融合结果为简单拼接）。

## 八、给前端开发者的开始步骤建议

1. **先通读这份文档**，理解数据流
2. **启动项目**：`npm run dev`，浏览器打开 `http://localhost:3000`
3. **看看现有界面的问题**，确认需要改哪些地方
4. **设计稿/原型**：建议先用 Figma 或手绘确定布局方案
5. **优先改回答列**（`AnswerColumn.tsx`）—— 这是用户看得最多的地方
6. **Markdown 渲染**：把 `SimpleMarkdown` 替换为 `react-markdown`
7. **再改输入面板**和**侧边栏**
8. **最后**：动画、响应式、微交互

## 九、需要询问前端开发者的关键问题

这些是你们需要一起沟通确认的：

1. **设计风格**：想要什么风格？（极简、玻璃态、毛玻璃、新拟态……）
2. **响应式策略**：小屏幕时三栏怎么变？（折叠/切换/上下排列？）
3. **Markdown 方案**：自己写渲染还是用库？（推荐 react-markdown）
4. **主题色**：当前紫色主色调是否保留？还是要改？
5. **Logo/品牌**：需要什么品牌视觉元素？
6. **组件库**：用 shadcn/ui 这类现成组件库还是全手写？
