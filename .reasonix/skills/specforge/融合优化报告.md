# SpecForge × AI 问答竞技场 — 双模型对话融合优化报告

> 本报告融合了 DeepSeek（附件对话）和 Kimi（网页链接对话）两段围绕 SpecForge 工作流和"AI 问答竞技场"产品的讨论，提取共识、标注差异、综合优化。

---

## 一、✅ 共识要点

以下为两个模型回答中高度一致的结论：

### 1. SpecForge 是目前最匹配的工作流
- 13 个 Skill 分三层（项目级 7 + 功能级 5 + 通用级 1），方法论成熟
- 核心机制：**边界守卫 GUARDRAILS** + **项目上下文协议 PROJECT-CONTEXT**
- V3 版本是关键迭代（从"操作手册"变为"角色设定"，引入 TDD + AC 全链路贯穿）

### 2. 推荐技术栈一致
| 层级 | 推荐方案 |
|------|---------|
| 全栈框架 | Next.js 14 (App Router) + TypeScript |
| AI 层 | Vercel AI SDK (`ai` 包) — 统一多厂商流式接口 |
| 样式 | Tailwind CSS + shadcn/ui |
| 拖拽布局 | react-resizable-panels |
| 数据库 | SQLite + Prisma（个人工具，零配置） |
| 部署 | Docker (Node 20 Alpine) |

### 3. 混合接入模式是独特的差异化
- API 接入：便宜/免费的模型（DeepSeek, Kimi, 智谱等）
- 网页自动化：贵的模型（GPT-4, Claude Opus, Gemini Pro）通过 Playwright 用免费网页版
- 架构上：API 模型用 `streamText()` 直接调，网页模型用 Playwright 注入 Cookie + 抓取 DOM

### 4. 核心产品定位一致
- 开源、自托管、个人用、问答式
- AI 裁判评分 + 纠错融合
- 面板式工作台（左输入 / 中多列回答 / 右评分+融合）

## 三、🔧 优化建议清单

基于上述分析，以下是针对 SpecForge 工作流和"AI 问答竞技场"产品的优化建议：

### 3.1 SpecForge Skill → Reasonix 适配优化

| 问题 | 建议 |
|------|------|
| 缺少 `runAs` 元数据 | 对 `bugfix-workflow` 和 `feature-implementation` 添加 `runAs: subagent`，隔离执行上下文 |
| 缺少 `allowedTools` | 为编码类 Skill 限制工具集（如只允许 read_file/write_file/bash），防止越界操作 |
| 路径引用不一致 | V3 部分 SKILL.md 中引用 `spec/`（不带 s） vs `specs/`（带 s），统一为 `specs/` |
| PROJECT-CONTEXT 路径硬编码 | 部分文件引用路径为相对路径，在 Reasonix 多项目场景下可能导致找不到文件 |
| 缺少 assets 模板 | 原仓库引用了 `assets/*.md` 模板文件，如果模板缺失 Skill 无法正常工作 |

### 3.2 AI 问答竞技场产品优化

| 方面 | 建议 |
|------|------|
| API 接入层 | 优先支持 OpenAI-compatible 接口（DeepSeek/Kimi/智谱/Moonshot），再扩展 Anthropic/Google 原生 SDK |
| 网页自动化 | MVP 阶段用本地 Playwright，后续迁移到 Browserbase/Stagehand 解决反爬虫 |
| 评分模型 | MVP 阶段一模型两用（GPT-4o 同时评分+融合），后续可拆分为评分（轻量）和融合（强模型） |
| 对话历史 | 默认全量存档（问题+各模型原始回答+评分+融合），SQLite 本地存储空间不是问题 |
| 前端状态 | 严格按照 DeepSeek 设计的状态机：idle → streaming → judging → fusing → complete |
| 限流保护 | 添加 API 调用并发控制和超时处理（单模型 45s 超时） |

### 3.3 流程层面优化

| 问题 | 建议 |
|------|------|
| "串行效率低"痛点 | SpecForge 作者坦诚串行开发慢。可引入 Reasonix 的 `task` 工具做并行子代理 |
| 审核机制缺失 | 添加审检查点 Skill（Audit Guard），在关键节点做质量门禁 |
| 文档同步问题 | 利用 Reasonix 的 `review` 工具在代码变更后自动触发文档同步检查 |
| 模型选择 | SpecForge 的 Skill 在执行时可利用 Reasonix 的 model routing，把不同 Skill 路由到最适合的模型 |

## 四、📋 综合产品定义（AI 问答竞技场）

### 产品名称
**AI 问答竞技场**（ArenaQA）

### 一句话定义
一个开源、自托管的个人 AI 问答工作台，将同一个问题并发发送给多个模型，由 AI 裁判自动评分并融合出最佳答案。

### 核心价值
- 打破信息茧房：同一个问题，看不同模型的回答角度
- 质量有保障：AI 裁判评分 + 纠错融合，不依赖用户判断
- 成本可控：API 调低价模型 + 网页免费蹭高价模型

### 目标用户
- 个人开发者 / 独立开发者（核心）
- AI 爱好者和研究者
- 需要频繁对比不同模型输出质量的内容创作者

### 核心工作流

```
用户输入问题
  │
  ▼
并发请求多个模型 ──────────────────────────────┐
  ├── API 模型（DeepSeek/Kimi/智谱等）          │ streamText()
  ├── 网页模型（GPT-4/Claude 等 via Playwright） │ DOM 抓取
  └── ...                                      │
  │                                             │
  ▼                                             │
全部回答收齐 ←──────────────────────────────────┘
  │
  ▼
AI 裁判评分（多维度：准确/完整/可操作/安全）
  │
  ▼
纠错融合（标记分歧、识别错误、生成综合答案）
  │
  ▼
面板展示：原始回答（多列） + 评分表 + 融合答案
```

### 关键设计决策（来自 DeepSeek）

**SSE 事件流**（7 种事件）：
1. `chunk` — 模型流式输出片段
2. `done` — 某个模型完成
3. `error` — 某个模型失败
4. `judge` — 裁判评分结果
5. `judge_error` — 评分失败
6. `fusion` — 融合答案
7. `complete` — 任务结束

**前端状态机**：
```
idle → streaming → judging → fusing → complete
```

**融合降级策略**：
- 所有模型一致 → 跳过差异分析，直接合并
- 仅 1 个模型成功 → 直接返回该模型答案
- 裁判模型失败 → 跳过评分，普通融合

## 六、📝 总结

两份对话从不同角度完整覆盖了 SpecForge 工作流和 AI 问答竞技场产品：

- **DeepSeek** 提供了产品的完整技术蓝图（API、状态机、数据模型、竞品分析），适合直接作为开发参考
- **Kimi** 提供了 SpecForge 的方法论全景和核心理念（文档驱动、阶段分离、引导式交互），适合理解"为什么这样设计"

融合后的最大价值在于：**方法论（Kimi）指导方向，技术蓝图（DeepSeek）指导落地**，两者结合可以立即开始 AI 问答竞技场的开发工作。
