# SpecForge V3 发布：功能级 Skill 的一次彻底重做

## 前言

之前发布了 SpecForge V2 版本，感谢大家的支持。这段时间一直在用这套工作流实际开发项目，用到最后发现了一个根本性的问题，V2 的 Skill 太像操作手册了，人读着费劲，AI 执行起来又太僵硬。于是花了不少时间，对功能级的五个 Skill 进行了近乎完全的重做，形成了 V3 版本。

项目级 Skill 本次没有更新，这次更新完全集中在功能级，也就是你日常开发中使用频率最高的部分。

---

## 一、更新了什么

### 1. 设计理念变了

V2 的 Skill 本质上是操作手册，步骤一做什么、步骤二做什么，写了九步，AI 从头到尾走一遍。对话格式也是预设好的，选项 A 怎样、选项 B 怎样、推荐哪个，不管什么场景都是这套格式。同一条规则还在好几个地方重复写，措辞还不一样。

V3 换了个思路。不再一步步教 AI 做什么，而是告诉它：你是什么角色、你要关注哪几个方面、你的底线在哪。至于具体先做什么后做什么、怎么跟用户沟通，让它自己判断。

简单说就是从「操作手册」变成了「角色设定」。

### 2. 编码阶段引入了 TDD

V2 的编码是按方案写代码，写完检查。V3 改成了 TDD，也就是 Red-Green-Refactor 循环：先写一个会失败的测试，再写刚好能让测试通过的代码，最后在测试保护下重构。

TDD 的影响不止于编码阶段。因为编码时必须先写测试，AI 就必须知道测什么。测什么来自验证标准，验证标准来自技术方案，技术方案来自需求。这意味着前面每个阶段的 AI 都会自动把精度拉上去，用户不需要操心这些，你说「我要个评论功能」就够了，AI 会在需求阶段通过对话帮你补全细节，在技术方案阶段自己把接口定义写清楚，在任务规划阶段自己把验证标准写具体。

换句话说，不是要求用户写得更精确，而是 TDD 倒逼 AI 在每个阶段都不敢糊弄，因为它知道最后要靠这些信息来写测试。

### 3. AC 贯穿了整条链路

AC 就是验收标准，V2 里也有，但各阶段各写各的，没有严格的对应关系。V3 要求每条 AC 必须在后续每个阶段都能找到落点。

举个例子，需求阶段产出了一条 AC：

> AC-001: Given 用户已登录，When 发布评论，Then 评论出现在列表顶部

那后续每个阶段都必须有它的着落：技术方案里要写清楚这条 AC 用什么接口实现、数据怎么存、排序逻辑是什么；任务规划里要有具体的任务来落地它；编码时要有测试来证明它确实做到了。

如果某条 AC 在技术方案的覆盖汇总表里找不到对应的设计点，说明有需求被遗漏了。如果任务规划里没有任务覆盖某条 AC，说明有东西没人做。每个阶段都在检查上一个阶段有没有漏东西，链路本身就是检查机制。

### 4. 输出模板大幅精简

V3 对模板做了大幅精简：

| 模板 | V2 行数 | V3 行数 | 缩减幅度 |
|---|---|---|---|
| 任务规划模板 | 303 行 | 84 行 | -72% |
| 阶段完成报告模板 | 178 行 | 55 行 | -69% |
| 功能迭代模板 | 190 行 | 126 行 | -34% |

### 5. 各 Skill 的具体变化

| Skill | V2 → V3 关键变化 |
|-------|-----------------|
| 需求澄清 | 四步线性问卷 → 五维自然对话，去掉预设提问模板 |
| 技术方案 | 九步线性流程 → 四维并行思考，规则去重 |
| 任务规划 | 测试是独立任务 → 测试融入 TDD 执行，新增通俗解释字段和验证标准质量要求 |
| 编码实现 | 写代码+检查 → 强制 TDD，去掉机械确认仪式 |
| 功能迭代 | 固定选项分类 → 自然对话 + 70% 阈值判断 |

## 二、完整项目流程教程

```
你的想法：我想给文章列表加个搜索
  │
  ▼
① feature-requirements-clarification   产品搭档，五维对话
   产出：搜索功能.md（含 AC）
  │
  ▼
② feature-tech-design                  技术搭档，四维设计
   产出：搜索功能_技术方案.md
  │
  ▼
③ feature-task-planning                技术主管，原子拆分
   产出：搜索功能_任务规划.md
  │
  ▼
④ feature-implementation               开发者，TDD 执行
   产出：代码 + 测试 + 完成报告
  │
  ▼
功能上线
  │
  需求变更？
  ▼
⑤ feature-evolution                    产品架构师，影响分析
   产出：变更任务计划 CR-001
  │
  ▼
回到 ④ 执行变更任务
```

贯穿整条链路的是 AC：需求产出 AC，技术方案映射 AC，验证标准对应 AC，TDD 测试验证 AC。任何一环断裂都会暴露。

---

## 三、写在最后

这次更新的本质，是从把 Skill 当操作手册变成了把 Skill 当角色设定。

旧版试图控制 AI 的每一个动作，结果是限制了它的能力。新版告诉 AI 它是谁、关心什么、底线在哪，然后让它在框架内自由发挥。

告诉 AI 要成为什么样的人，而不是规定它每一步要做什么动作。
